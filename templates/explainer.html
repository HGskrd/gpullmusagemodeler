TL;DR — what this is

A roofline-style planner for vLLM deployments. You describe the hardware you have (GPUs, memory caps), the models you want to serve (with precision and parallelism), and the demand — a portfolio of use cases, each with a token demand, difficulty / quality SLO, WTP ceiling, capability requirements, and an input/output length preset. The planner estimates peak throughput and per-user latency under vLLM's continuous batching, and projects how that demand routes across your cluster versus a corporate cloud fallback.

It's a closed-form estimator, not a simulator. Numbers are for sizing conversations (GPU count, precision, TP/PP/DP), not SLA guarantees. Calibrate against vLLM's runtime metrics on representative workloads before trusting fine detail.

Quickstart

  1. Add a GPU pool. Click + Add GPU, pick a card (e.g. MI355X, H100), and choose how many are in the pool.
  2. Add one or more models. Click + Add model and pick from the catalog. The planner auto-chooses a GPU count and a TP/PP/DP topology that fits.
  3. Shape the workload. Open the Use cases panel and pick each use case's Input shape / Output shape presets (Chat, Long doc, Code, Classify, RAG). The planner's aggregate prefill/decode distribution is a demand-weighted blend of those choices. Optionally set a prefix-hit rate in the panel below.
  4. Read the chart. Switch between User Pareto (tok/s per user vs concurrency) and Processing Pareto (requests/sec vs batch size).
  5. (Optional) Open the Internal market panel below the chart to see how demand from projects routes onto this deployment vs cloud.
  6. (Optional) Click ⊞ Duplicate To Compare to fork the current config into a B-side and edit it independently for A/B comparisons.

The charts

The main chart below the config panels has two modes, selected via the tabs beneath it:

User Pareto

Plots tok/s per user vs concurrent users (both log-scale). This is the latency view: adding users drops per-user throughput until the system is bandwidth- or KV-bound. The curve flattens where each model is memory-bandwidth limited on decode, and drops off a cliff when the KV cache runs out of room (OOM — shown as a break in the line). Hover any point to see total tok/s and ms/token for that operating point.

Processing Pareto

Plots requests/sec vs batch size (both log-scale). This is the throughput view: how many requests/sec the deployment sustains end-to-end, including prefill and decode together (with their interference blend). Useful for sizing batch-oriented workloads where per-user latency matters less than aggregate throughput.

Each deployed model contributes one line. In compare mode (A/B), panel B's lines are drawn dashed and suffixed with (B). A legend under the config area shows each line's model, phase (P/D), topology, and precision.

Configuring hardware

GPU pools

Each pool is a bag of identical GPUs. The card header shows the GPU type, memory bandwidth, vendor, and how many of the pool's GPUs are still free (not claimed by any model). Use the − / + buttons to resize the pool; removing GPUs that are already assigned will trim the neediest model's count down automatically.

The bandwidth shown in the card header is the vendor's published peak memory bandwidth. For a few device families with sparse public inference disclosure, the planner may use a more conservative sustained-bandwidth proxy internally.

TCO $/GPU·hr

Amortized total cost of ownership per GPU-hour: hardware capex spread over its useful life, plus power, cooling, datacenter, and ops. When set (non-zero), it powers the daily cluster cost and the coverage / margin readout in the internal-market panel. Leave it at 0 and those readouts are suppressed.

Memory cap

gpu_mem_util
Requested fraction of device memory (HBM or unified memory) the runtime can use (0.50–0.98). Weights and profiled non-KV runtime memory are subtracted from this to determine the KV budget.
Profiled non-KV
GB reserved per GPU for activations, CUDA graphs, NCCL scratch, sampler state, and other runtime memory not captured by the static weight footprint. Larger → less KV headroom.

Configuring models

Adding a model

+ Add model opens a picker grouped by category (dense, MoE, reasoning, …). On add, the planner picks the smallest GPU count on the best-fitting pool that has a valid topology in BF16 (falling back through the lower-memory precision options if BF16 doesn't fit), then tunes TP/PP/DP to whichever valid strategy maximizes peak throughput.

Per-model controls

Precision
Per-model weight precision. FP8 uses 1 B/param and FP8 KV cache. NVFP4 and MXFP4 use E2M1 4-bit weights plus block-scale metadata, while keeping KV cache at FP8 for a less optimistic serving estimate.
On <pool>
Visible when you have more than one GPU pool. Move the model between pools.
GPUs
How many GPUs from the chosen pool this model uses. Each +/– retunes parallelism.
P row
Valid prefill topology buttons. Labels read TPx·PPy·DPz, omitting dims that equal 1. A highlighted button is the active choice.
D row
Same, for decode. Prefill and decode can use different topologies; the planner models the handoff via the P/D interference blend.

Topology glossary

TP (tensor parallel)
Shards each layer's weights across TP GPUs. All-reduces every layer — latency-sensitive, prefer within a single node.
PP (pipeline parallel)
Stages the model across PP GPUs; each micro-batch flows through stages. Cheaper cross-node than TP, but adds pipeline bubbles.
DP (data parallel)
Replicates the model across DP independent instances; each serves its own batch. Throughput scales roughly linearly with DP.

The planner warns when TP spans nodes (cross-node all-reduces tank bandwidth) and when MoE multi-node expert traffic or long-context KV sharding isn't modeled — treat those views as optimistic.

Memory readouts

Each model card shows two lines after topology: P ... GB effective KV · ~N prompts at L tok and D ... GB effective KV · ~N slots. These are the prefill- and decode-side KV budgets after subtracting weights and profiled non-KV, plus the batch-size or concurrency that budget supports at the current workload lengths.

Workload shape

Per use case: Input / Output shape

Each use case card owns its own Input shape and Output shape preset (Chat, RAG, Long doc, Code, Classify). The aggregate prefill/decode distribution consumed by the rooflines is a demand-weighted blend across all active use cases — weight each bucket by project.tokens_day. Change a preset and the planner retunes every model's topology for the new averaged sequence length.

Before, the Input / Output length distribution lived in two standalone slider cards. They're gone now — the same information is split across use cases, which also lets you explain why a particular distribution applies (classification has different shapes than deep research).

Quality and token efficiency — why the right model matters

Each use case declares a Difficulty (task hardness, 0..100) and a Quality SLO (minimum success rate). The planner inverts success_rate = sigmoid(k·(quality − difficulty)) to compute a required model quality floor, shown inline on the card (Needs model quality ≥ 0.xx). Any deployed model whose quality falls below that floor is rejected at routing time. Model quality is calibrated from Artificial Analysis' Intelligence Index into the 0.30–0.95 range.

Each model also has a token efficiency multiplier (η). A model that uses ~10M output tokens on the Intelligence Index reference has η = 1.00×; a more verbose model (Nemotron Nano, ~140M tokens for the same benchmarks) has η ≈ 0.07×, meaning it costs roughly 10× more tokens than Gemma 4 31B (≈ 1.4×) to answer the same question. The internal market divides on-prem $/M by η to compute an effective $/M, so a verbose model that's cheap per token often loses to a compact one with the same headline price.

Both quality (Q) and efficiency (η) are visible on every deployed model card — the two numbers sit right below the weight footprint. Use them to reason about which use case will route where before running the market.

Prefix reuse

Prefix hit rate models vLLM's Automatic Prefix Caching: the fraction of prompt tokens served from cache instead of re-prefilled. It only affects prefill-sensitive metrics; pure-decode and aggregate Pareto curves are unchanged. Effective prefill length = input_len × (1 − hit_rate).

Efficiency parameters

Under ⚙ Efficiency parameters in the cluster panel. These are the knobs that make the roofline match reality; the defaults are sane for current vLLM builds, but you'll want to calibrate per-workload.

Prefill-only

BW efficiency
Fraction of peak memory + collective bandwidth achieved during prefill.
Compute efficiency
Fraction of peak FLOP/s achieved during prefill.
Overhead factor
Flat runtime overhead applied to prefill.
Paged KV overhead
Tax for KV paging during prompt ingestion.
AR overlap
Fraction of all-reduce hidden behind prefill compute.

Decode-only

BW / Compute / Overhead / Paged KV / AR overlap
Same meaning as prefill, but applied to the decode loop.
Sched budget
Approximate scheduler working-set token budget before decode efficiency degrades. Caps the number of concurrent decode slots per replica.

Shared

KV slack reserve
Residual KV loss after allocation (fragmentation, edges).
MoE imbalance
Multiplier penalizing expert skew and routing tails. 1.00× = perfect balance.
P/D interference
Blend between fully overlapped prefill+decode (0%) and fully serialized (100%). Higher means prefill starves decode and vice versa.

Compare (A/B)

Click ⊞ Duplicate To Compare (top-right) to fork the current state into a second panel. Edits on either side are independent. Both configs render on the same chart — B's curves are dashed — and both appear in the internal-market panel side by side. Click ✕ Close Compare to collapse back to a single config.

Typical use: clone the deployed config, change one thing (precision, GPU count, an efficiency knob), and eyeball whether it moves the Pareto frontier or shifts internal-market fates.

Internal market (revenue projection)

A demand-side layer stacked on top of the throughput model. You describe projects (workload streams with a price ceiling and a quality requirement); the planner routes those streams across your deployed models, spills excess to cloud, and tells you what happens to the rest.

Use cases — the demand side

Demand (tok/day)
Daily token volume the use case brings.
WTP ceiling ($/M)
Willingness-to-pay per million tokens. If effective on-prem price exceeds this, the use case prefers cloud; if cloud also exceeds it, the work is destroyed. Effective $/M divides raw $/M by the routed model's token_efficiency, so a verbose model effectively costs more than its headline price.
Difficulty
Task difficulty ∈ [0, 1], paired with each model's benchmark-anchored quality via success_rate(quality, difficulty) to decide eligibility.
Quality SLO
Floor on success_rate. Candidate models below this are rejected. The card derives a Needs model quality ≥ q line inline — invert sigmoid(k·(q − difficulty)) = slo to get the minimum eligible quality.
Input / Output shape
Per use case preset for prompt / output length distribution. The aggregate distribution consumed by the rooflines is the demand-weighted blend across all use cases' presets.
Requires (capabilities)
Hard capability gates (tool use, vision, long-context, …); a model must supply ALL listed capabilities to be eligible.
Latent demand / unlock
Hidden pool that only materializes when the cheapest viable on-prem $/M falls at or below the unlock threshold. All-or-nothing per routing pass.

Ceiling order: higher-WTP projects win capacity first. Project presets seed realistic difficulty/demand/WTP combos (classification, chatbot, coding, research).

Where demand ends up — the fate bar

  • Served — routed to one of your deployed models; counted at internal $/M.
  • Spilled — a compatible model is deployed, but you're out of GPU capacity, so overflow goes to cloud.
  • Leaked — no deployed model matches (capabilities, SLO, or pricing beats internal), so the project uses cloud directly.
  • Destroyed — even the cheapest compatible cloud model is above WTP, or no cloud model meets the SLO. The work is shelved.

Corpo cloud catalog

Procurement decides which cloud models the gateway exposes. The active catalog appears as tags; switching between current and advocated shows the demand-destruction impact of expanding vendor access. Reference cloud $/M for each project is the cheapest compatible model in the active catalog.

Supply — deployed models table

One row per deployed model: quality, token efficiency, capabilities, GPUs, daily-token capacity, demand placed, utilization, and on-prem $/M. Tags: SATURATED means all capacity consumed (adding GPUs would stop spill); IDLE means no project routed here (quality doesn't meet SLOs, or $/M exceeds every ceiling).

Advanced — day shape, night batching, workload

Day shape
Hourly demand profile (e.g. workday, flat, nocturnal). Changes the peak- to-mean ratio; sustainable daily tokens = peak RPS × 86400 × tokens/req ÷ peak-factor.
Night batching
Flattens peak for batch-eligible projects by offering a discount overnight. Demand shifts via an elasticity parameter.
Workload readout
Average in/out lengths and active prefix-hit rate used by the market pricing.

Glossary

Roofline
A closed-form bound on throughput from either bandwidth (KV/weight fetches) or compute (FLOPs), whichever binds first.
Continuous batching
vLLM's scheduler: decode slots are refilled request-by-request rather than in fixed micro-batches, keeping the GPU busy across uneven request lengths.
KV cache
Per-token attention state that must live in accelerator memory for the duration of a request. KV per token depends on model dimensions and precision.
MLA
Multi-head Latent Attention (DeepSeek-style): compresses the KV cache, so decode concurrency is far higher than vanilla attention for the same memory budget.
MoE
Mixture-of-Experts. Total params are large but only a subset activates per token. Throughput uses active params for compute, plus an imbalance multiplier for the routing tail.
Prefill / Decode
Prefill = ingesting the prompt (compute-bound, wide). Decode = generating output tokens one step at a time (bandwidth-bound, many concurrent).
Effective $/M
Internal $/M divided by a per-model token-efficiency multiplier. Models that think more or are more verbose cost more per unit of delivered value.
